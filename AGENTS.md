# AGENTS.md
# こころの羅針盤 開発憲章

## 1. Purpose

「こころの羅針盤」は、心を整理するための **AI内省サービス** である。

このプロダクトは以下ではない。

- 雑談用AIチャット
- 占いアプリ
- 即答型の悩み解決サービス
- 強い助言を与えるメンタルコーチ

このプロダクトが提供するものは、**内省体験（Self Reflection Experience）** である。

ユーザーがこのサービスを通じて行うことは以下。

- 悩みを言葉にする
- 心の中を整理する
- 気づきを得る
- 記録を残し、あとで振り返る
- 自分の人生の流れを見つめる

AI名は **ソラ**。

---

## 2. Product Philosophy

このサービスは「答えを与える」ためのものではない。  
**気づきを生むためのもの** である。

AIは正解を提示しすぎてはいけない。  
ユーザー自身が自分の言葉、自分の感情、自分の答えに近づくための補助を行う。

重要なのは以下。

- 共感
- 整理
- 問い
- 気づき
- 振り返り

避けるべき方向性。

- 断定的な助言
- 説教
- 一方的な解決策の提示
- 占いっぽい演出
- 過度にスピリチュアルな断言
- チャットのテンポ感を優先する設計

---

## 3. Current Tech Stack

- Next.js (App Router)
- TypeScript
- Tailwind CSS
- OpenAI API
- localStorage

Deployment:

- GitHub
- Vercel

---

## 4. Technical Constraints

このプロジェクトでは、現時点で以下を行わない。

- 認証
- Stripe
- Supabase
- 外部DB
- 管理画面
- 分析基盤
- 大規模リファクタ
- アーキテクチャ全面変更

データ保存は **localStorage ベースを維持** すること。

AIエージェントは、勝手に永続化方式を変更しないこと。  
外部サービス追加を前提に提案しないこと。  
MVPの軽量性を壊さないこと。

---

## 5. Development Goal

現在の状態は「公開できるMVP」。

目標はこれを、

**公開できるMVP** から  
**完成度の高いAIプロダクト**

へ引き上げること。

ただし、そのために大きく作り変えるのではなく、以下を守る。

- 既存MVPを壊さない
- 小さく改善する
- UI/UXを優先する
- 相談体験を自然にする
- 内省体験を深める

---

## 6. Development Priorities

優先順位は以下。

1. UX
2. 内省体験の自然さ
3. UIの静けさと上品さ
4. 読みやすさ
5. 安定性
6. 最小限の実装コスト

優先度が低いもの。

- 大きな機能追加
- 新技術導入
- 大規模な構造変更
- 見た目だけの派手な演出

---

## 7. UI Worldview

UIの世界観は以下。

- 静か
- やさしい
- 上品
- 余白が多い
- 白ベース
- 薄紫アクセント
- 心理 / 内省 / スピリチュアルの中間
- ただし「占いUI」には寄せない

避けるべきUI。

- 派手
- 情報量が多すぎる
- 装飾過多
- 占いアプリ風
- チャットアプリ風
- SNS風
- ポップすぎる演出
- 強すぎるグラデーションや装飾
- 不必要なアニメーション

デザイン判断に迷った場合は、**静けさと余白を優先** すること。

---

## 8. Core Consultation Screen Structure

相談画面の理想構造は以下。

1. テーマ選択
2. 共感
3. 質問
4. 入力欄
5. 今日の小さな気づき
6. 未来のあなたからの言葉
7. 次の問い
8. 心の状態メーター
9. ログ導線
10. フィードバック導線

この構造は重要であり、AIエージェントは勝手に並び順や主従関係を変えないこと。

主役は以下の2つ。

- 質問
- 入力欄

この2つが体験の中心である。  
UI改善を行う場合も、この2つを最優先で見やすくすること。

---

## 9. UX Principles

相談体験は「書かされる体験」ではなく、  
**自然に言葉が出てくる体験** を目指す。

そのため、以下を重視する。

- どこを見ればいいかが一瞬でわかる
- どこに入力すればいいか迷わない
- AIの返答が読みやすい
- 圧がない
- 考える余白がある
- 少し安心できる
- 書いてもいいと思える

改善対象として特に重要なもの。

- 入力欄の見つけやすさ
- 質問文の見やすさ
- AIレスポンスの可読性
- ログ画面の振り返り体験化

---

## 10. Sora Persona

AI「ソラ」は以下の人格を持つ。

- やさしい
- 落ち着いている
- 判断しない
- 押し付けない
- 少し静か
- 受容的
- 上品
- 人間の気づきを邪魔しない

ソラの返答は以下を中心に構成する。

- 共感
- 問い
- 小さな気づき
- 未来へのやわらかい視点

避ける表現。

- 強い断定
- 説教
- 命令
- 正論で押し切る言い方
- スピリチュアル断言
- 過度な励まし
- テンションの高すぎる表現
- 軽すぎるカジュアル表現

---

## 11. Logging Philosophy

相談ログは単なる履歴ではない。  
**人生の記録** として扱う。

ログ画面の役割は以下。

- 振り返る
- 自己理解を深める
- くり返し現れるテーマに気づく
- 人生の流れを見る

そのため、ログUIは「保存一覧」ではなく、  
**振り返り体験** に近づけること。

---

## 12. UI Copy QA

ユーザー向けの表示文言を変更した場合は、実装完了前に必ず文言チェックを行うこと。

チェック対象:

- 見出し
- ボタン文言
- 補助文
- プレースホルダー
- エラーメッセージ
- 完了メッセージ
- 問い文

確認項目:

- 誤字脱字がないか
- かなの打ち間違いがないか
- 漢字変換ミスがないか
- 読んだときに引っかからない自然な日本語か
- 丁寧さのトーンが揺れていないか
- 説明しすぎていないか
- ソラの静かな世界観に合っているか
- AIっぽい不自然さが残っていないか

特に問い文は重点確認すること。
意味が通るだけではなく、日本語として自然で、静かに読めるかまで確認する。

実装ルール:

- 新しい文言を追加・変更したら、変更箇所の周辺文言までまとめて見直す
- 一文だけ直して終わりにしない
- 「ひとばん」のような、意味は推測できるが不自然なかなの取り違えを見逃さない
- 少なくとも 1 回はスマホ幅で実際の表示を目視確認する

---

## 12. Feedback Philosophy

フィードバック導線は単なるアンケートではない。  
**ユーザーと一緒に育てるための共創導線** である。

表示は、お願い口調よりも、  
「一緒に良くしていく」ニュアンスを優先すること。

Googleフォーム導線は維持してよいが、  
目立ちすぎず、自然に置くこと。

---

## 13. Code Change Policy

AIエージェント / Codex は以下を守ること。

1. 既存機能を壊さない
2. 小さく変更する
3. UI構造を崩さない
4. 新しい依存を安易に追加しない
5. ファイル構造を勝手に大きく変えない
6. 実装理由を説明できる変更だけを行う
7. 一度に多くを触りすぎない
8. ローカル状態管理と localStorage 前提を守る

特に禁止。

- 無断の大規模リネーム
- 無断の責務分割
- 無断のディレクトリ再編
- 無断の状態管理ライブラリ導入
- 無断のDB化
- UI全入れ替え
- チャットアプリ化

---

## 14. UI Change Policy

UI変更は **体験改善のためにのみ許可** する。

変更してよいもの。

- 余白
- 配色の微調整
- タイポグラフィ
- カードの見やすさ
- セクション間の整理
- 入力欄の見つけやすさ
- ボタンの自然な強調
- モバイル時の視認性
- AIレスポンスの読みやすさ

原則として避けるもの。

- レイアウト全面変更
- 画面構成の組み替え
- 主従関係の逆転
- チャットUI化
- 演出過多
- 不必要なアイコン追加
- 視線誘導を壊す装飾

---

## 15. AI Response Presentation Policy

AIレスポンスは、単なるテキストの塊として表示しない。  
**受け取りやすい内省の断片** として見せる。

表示設計で重視すること。

- セクションが分かれていること
- 一文が長すぎないこと
- 行間が苦しくないこと
- 主役と補助が視覚的にわかること
- 特別な一言は少しだけ印象的に見せること

特に以下は読みやすさが重要。

- 共感
- 質問
- 今日の小さな気づき
- 未来のあなたからの言葉
- 次の問い

---

## 16. Input Experience Policy

入力欄は「フォーム部品」ではなく、  
**言葉を置ける場所** として設計する。

重視すること。

- 見つけやすい
- 押しつけがましくない
- 書き始めやすい
- 緊張を生まない
- 余白がある
- モバイルで窮屈にならない

プレースホルダや補助文は、  
「うまく書かなくてよい」という安心感を補助するものが望ましい。

---

## 17. Working Style for Codex

Codex は必ず以下の順で動くこと。

1. plan
2. 実装
3. 修正

いきなりコードを書き始めないこと。  
まず「何を・なぜ・どこまで触るか」を簡潔に示すこと。

### Required Output Style

作業開始時は以下の形式を推奨する。

- 目的
- 変更対象
- 変更内容
- 壊さないための確認点

実装後は以下を示す。

- 何を変更したか
- どこに影響するか
- 確認してほしい点

---

## 18. Scope Control

1回の作業で触る範囲はできるだけ小さくすること。

望ましい単位。

- 1画面
- 1コンポーネント
- 1体験課題
- 1UIテーマ

望ましくない単位。

- サービス全体の一括改修
- 複数画面の同時刷新
- UIとロジックの全面整理
- 今後必要になる前提の先回り実装

---

## 19. Preferred Improvement Areas

今後の改善で優先してよい領域。

- 相談画面のUI完成度向上
- 質問と入力欄の主役化
- AIレスポンスの読みやすさ改善
- ログ画面の振り返り体験化
- モバイル時の入力体験改善
- 世界観の統一
- 小さな文言改善
- ローディングや保存通知の自然化

---

## 20. Final Rule

迷ったら、以下の順で判断すること。

1. その変更は内省体験を深めるか
2. その変更は静けさを守るか
3. その変更は入力しやすさを上げるか
4. その変更は既存MVPを壊さないか
5. その変更は小さく安全に実装できるか

このファイルの目的は、  
開発者 / Codex / AIエージェントが  
同じ思想で「こころの羅針盤」を育てることにある。
